Problem(s) Addressed

  • Netlify does not abide by the current engineering standards
    • Currently, the front-end is hosted on Netlify, which is not supported by current engineering standards.
    • connection to the repo using amplify
    • Possibly look into deploying the back end to AWS (stretch)

  • Dropbox switch over for media storage - from firebase
    • Dropbox is used by Wes currently by his company pretty heavily
      • would definitely be more of a hassle for Wes and his company to be converting their files from firebase to Dropbox
    • As a team, none of us have used dropbox in any of our projects - plenty of research to be done
    • How to implement? Users login to their current account, or an App account? - Wes has a shared folder with all current employees at the moment
      • are there restrictions on whether we can have one account linked to everyone using the app?
        • what about when there are other organizations that join schematic capture?
    • Dropbox documentation stinks... - Where to find the answers to all our questions?

  • Okta switch over for authorization and security - from firebase
    • Currently, the front-end uses Firebase for authentication, which is not allowed by current engineering standards.
    • No user types in current state of app - didn't want to implement them when using firebase
      • Do not allow for technicians to delete photos, update the schematics, edit the parts list, assign jobs to technicians, create clients, create projects, invite users
      • Allow for technicians to access all jobs in their organization (this may not have any real distinguished features until later RC's), access jobs assigned to them, mark parts that should be updated (ooo), upload images to a jobsheet that they're assigned to, and that does not have an image that is currently uploaded for that specific part
      • Admins have access to everything - assume there is only one admin at this point

Solution

  • Figuring out how to use AWS for our deployed website
    • Connect DNS to AWS rather than Netlify
    • Deprecate current Netlify deployment
    • AWS is an optional course given to us next week in labs
      • Amplify short hour demo will also be a great help - video to come

  • Figure out the capability of Dropbox, and whether or not we will be able to work with it in this in the long run
    • Setup test Dropbox
    • Find out ...
      • How to upload a picture to dropbox
      • How to access a picture from Dropbox - will this bring some security risks to our app?
        • if it does, will that lose integrity of what we are trying to accomplish? We feel like a big point of this app is to see these photos that are being uploaded. Would like users to access them even after uploaded
      • If it's possible to create folders within Dropbox - maybe on creation of a new project we could create a folder in Dropbox for organization - no need for Wes to do it manually
        • how to store path to this folder for when we want to post / access photos here
      • back end ⇒ Dropbox or Dropbox ⇒ back end for storage?
    • Wes's current Dropbox? Create new Dropbox? - currently a shared folder among all current employee's Wes organizes uploads after the fact.
    • Possible later question ⇒ how to setup a new dropbox area for separate organization from Wes and Alloy Technologies

  • Figure out the capability of Okta, how to implement
    • Create Okta account for Schematic Capture
    • Create test project using Okta to ease integration into the main application.
      • find the endpoints that we need to hit so there is no front end messiness - only back end deployment (clean)
    • Create user groups corresponding to user types in the Schematic Capture application.
    • Change authorization endpoints and middleware to utilize Okta instead of Firebase.

Objectives Supported

  • Switch over to the more reputable deployment service in AWS from netlify to ensure as frequent up time as possible limiting factors causing faulty connections.
  • Adopt current Labs 22 engineering standards in supporting Okta authentication over Firebase
  • Switch over all file sharing to Dropbox

Key Results

  • Code base conforms to current engineering standards by 100% (authentication, and deployment method)
  • Implement Dropbox to allow for current employees of Alloy Solutions to maintain current workflow
  • Eliminate Firebase, Netlify, and Google login from the current code base.

End Results

  • AWS has increased the speed at which our team is able to tackle new problems, while also increasing the speed of normal usage of the application as a whole. No downtime has been recorded.
    • We have needed separately deployed branches - AWS makes this a breeze compared to netlify. These deployments are also quicker than past deployments as well further increasing productivity
  • Current workflow of employees at Wes's company will not change at all - their Dropbox related work has stayed completely the same besides the fact that their uploading will be done through our application as opposed to Dropbox itself. They will continue to use their own specific Dropbox, with shared folder from stakeholder Wes.
